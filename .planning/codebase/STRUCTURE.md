## Directory Purposes

**src/:**
- Purpose: All TypeScript source code (ESM format, `.ts` files)
- Contains: Commands, business logic, providers, utilities, tests
- Key files: `cli.ts` (entry), `add.ts` (main command), `agents.ts` (40+ agents)
- Build step: `pnpm build` → obuild compiles to `dist/` and `bin/cli.mjs`

**src/providers/:**
- Purpose: Abstract skill fetching from different hosts (extensible)
- Contains: `HostProvider` interface, concrete implementations
- Key files: `wellknown.ts` (Mintlify registry), `market.ts` (internal Market), `cos.ts` (COS storage)
- Usage: Add command resolves sources via `providers/registry.ts`

**bin/:**
- Purpose: Executable entry point for npm
- Contains: Single compiled file `cli.mjs` (shebang + bundled code)
- Generated by: `obuild` build tool (configured in `package.json` > build script)
- Used by: npm bin, PATH after `npm install -g`

**dist/:**
- Purpose: Compiled TypeScript (ESM)
- Contains: All `.ts` files compiled to `.js`
- Generated by: `obuild build` command
- Published: Included in npm package (see `package.json` > files array)

**scripts/:**
- Purpose: Build-time and maintenance scripts
- Key files:
  - `generate-licenses.ts`: Produces `ThirdPartyNoticeText.txt` (required for publication)
  - `validate-agents.ts`: Verifies agent definitions are complete
  - `sync-agents.ts`: Updates README with agent list

**tests/:**
- Purpose: Integration tests and edge-case validations
- Key tests: Path traversal prevention, fuzzy skill matching, source parsing, symlink fallback
- Run with: `pnpm test` (vitest runner)
- Separated from `src/` for slower/more isolated tests

## Key File Locations

**Entry Points:**

- `bin/cli.mjs`: Executable script (installed to PATH after `npm install -g`)
- `src/cli.ts`: TypeScript CLI source, routes commands, shows banner/help

**Configuration:**

- `package.json`: Dependencies (clack, picocolors, simple-git, xdg-basedir), scripts, version (1.4.3-bmc1.1.x)
- `tsconfig.json`: Target ES2022, module ESM, strict mode, source maps
- `vitest.config.ts`: Test runner, coverage, reporters
- `.husky/`: Pre-commit hooks (runs `prettier --write` before commit)
- `.prettierrc`: Code formatting rules (spacing, quotes, etc.)

**Core Logic:**

- `src/add.ts`: Main add command (~500 lines) - parse source, fetch skills, prompt, install, update locks
- `src/installer.ts`: Symlink/copy logic (~400 lines) - handles Windows, broken symlinks, self-referential links
- `src/agents.ts`: Agent definitions (~300+ lines) - AgentConfig for each agent, platform paths
- `src/skills.ts`: Skill discovery (~250 lines) - recursive SKILL.md search, frontmatter parsing
- `src/source-parser.ts`: URL/shorthand parsing (~200 lines) - GitHub, owner/repo, local paths, Market syntax

**Testing:**

- `src/*.test.ts`: Co-located unit tests (add.test.ts, cli.test.ts, list.test.ts, remove.test.ts, init.test.ts)
- `tests/*.test.ts`: Integration tests (path traversal, skill matching, source parsing)
- `src/test-utils.ts`: Shared test fixtures and helpers

**Locks & State:**

- `src/skill-lock.ts`: Global lock file (~250 lines) - `~/.agents/.skill-lock.json`, GitHub tree SHA, version tracking
- `src/local-lock.ts`: Project lock file (~250 lines) - `./skills-lock.json`, computed file hashes, deterministic sorting
- Both are JSON with version field for backwards-incompatible migrations

**Customizations (BMC Fork):**

- `src/branding.ts`: Branded URLs (SKILLS_SITE, SKILLS_API_BASE), telemetry enabled/disabled (default: off)
- `src/providers/cos.ts`: Tencent COS storage provider (BMC custom, not in upstream)
- `src/providers/market.ts`: Internal Skills Market provider (BMC custom, not in upstream)
- `src/telemetry.ts`: Anonymous tracking control (default disabled, can opt in)

## Naming Conventions

**Files:**

- Command files: `<command>.ts` (e.g., `add.ts`, `list.ts`, `remove.ts`)
- Test files: `<module>.test.ts` or `<module>.spec.ts` (co-located in `src/` or separate in `tests/`)
- Provider files: `<provider-name>.ts` in `src/providers/` (e.g., `wellknown.ts`, `cos.ts`)
- Utilities: Single-responsibility names like `installer.ts`, `skills.ts`, `agents.ts`

**Directories:**

- Command bundles: Flat in `src/` (no subdirectories per command)
- Feature areas: Explicit subdirectories (`src/providers/`, `src/prompts/`)
- Tests: Either co-located or in `tests/` for integration tests

**Functions:**

- Commands: `run<Command>()` prefix (e.g., `runAdd()`, `runList()`, `runFind()`)
- Options: `parse<Feature>Options()` pattern (e.g., `parseAddOptions()`)
- Internal: Lowercase camelCase (e.g., `sanitizeName()`, `discoverSkills()`, `getAgentBaseDir()`)
- Interfaces: PascalCase with `I` prefix discouraged (e.g., `AgentConfig`, `Skill`, `ParsedSource`)

**Variables:**

- Constants: UPPER_SNAKE_CASE (e.g., `AGENTS_DIR`, `CURRENT_LOCK_VERSION`)
- Colors/ANSI: CONSTANT naming (e.g., `RESET`, `BOLD`, `DIM`, `TEXT`)
- Temporary state: camelCase (e.g., `selectedIndex`, `debounceTimer`)

**Types:**

- Enums: None used (discriminated unions preferred)
- Union types: Named explicitly (e.g., `AgentType`, `InstallMode`, `ParsedSource['type']`)
- Generic parameters: Short uppercase (T, U, K, V as appropriate)

## Where to Add New Code

**New Command (e.g., `skills purge`):**

1. Create `src/<command>.ts` with:
   - `parse<Command>Options(args: string[]): <Command>Options` to extract flags
   - `run<Command>(args: string[], options: <Command>Options): Promise<void>` main logic
2. Add test file: `src/<command>.test.ts` with vitest suite
3. Wire into CLI: Add case in `src/cli.ts` switch statement (line 640-710)
4. Update help text: Add to `showHelp()` and `showBanner()` functions in `src/cli.ts`

**New Agent Support (e.g., `new-agent-x`):**

1. Add config to `agents` object in `src/agents.ts`:
   ```typescript
   'new-agent-x': {
     name: 'new-agent-x',
     displayName: 'New Agent X',
     skillsDir: '.newagent/skills',
     globalSkillsDir: join(home, '.newagent/skills'),
     detectInstalled: async () => existsSync(join(home, '.newagent')),
   }
   ```
2. Add to `AgentType` union in `src/types.ts` line 1-42
3. Update `scripts/validate-agents.ts` to include new agent
4. Run `pnpm run sync-agents` to update README

**New Provider (e.g., `HuggingFace` registry):**

1. Create `src/providers/<provider>.ts` implementing `HostProvider` interface (lines 38-75 in `types.ts`):
   - `readonly id: string`
   - `readonly displayName: string`
   - `match(url): ProviderMatch`
   - `fetchSkill(url): Promise<RemoteSkill | null>`
   - `toRawUrl(url): string`
   - `getSourceIdentifier(url): string`
2. Export from `src/providers/index.ts`
3. Register in `src/providers/registry.ts` (or inline in add.ts provider resolution)
4. Add tests in `tests/<provider>.test.ts`

**New Test (edge case):**

- Co-located: Add to existing `src/<module>.test.ts` if testing internal function
- Integration: Create `tests/<feature>.test.ts` for multi-module scenarios
- Use `describe()`, `it()`, `expect()` from vitest
- Fixture data: Reference `src/test-utils.ts` for common test agents, paths, skills

**Shared Utility (e.g., `sanitization.ts`):**

1. Create `src/<utility>.ts` with focused responsibility
2. Export named functions (avoid default export)
3. Add unit tests alongside: `src/<utility>.test.ts`
4. Import where needed via `import { fn } from './utility.ts'`

## Special Directories

**~/.agents/ (User home):**
- Purpose: Global skill installation directory
- Generated: No (created on first use)
- Committed: No (user-scoped, not in repo)
- Contents:
  - `~/.agents/skills/` - symlinks/copies to installed skills
  - `~/.agents/.skill-lock.json` - global lock file

**.agents/ (Project root):**
- Purpose: Project-scoped agent directories and skills
- Generated: No (created on `skills add` without `-g`)
- Committed: Yes (skills-lock.json should be checked in)
- Contents:
  - `.agents/skills/` - local installed skill copies/symlinks
  - `skills-lock.json` - local lock file (committed, minimal format for diffs)

**dist/ (Build output):**
- Purpose: Compiled TypeScript output
- Generated: Yes (by `pnpm build`)
- Committed: No (in `.gitignore`)
- Produced by: obuild tool (configured in `package.json` script)

**node_modules/ (Dependencies):**
- Purpose: npm packages
- Generated: Yes (by `pnpm install`)
- Committed: No (use lockfile instead)
- Lockfile: `pnpm-lock.yaml` (committed, managed by pnpm)

**.planning/codebase/ (GSD planning):**
- Purpose: Architecture and structure documents
- Generated: Yes (by `/gsd:map-codebase` command)
- Committed: Yes (tracked in Git)
- Contents: ARCHITECTURE.md, STRUCTURE.md, CONVENTIONS.md, TESTING.md, CONCERNS.md
